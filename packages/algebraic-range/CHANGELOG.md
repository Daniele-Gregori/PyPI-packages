# Changelog

All notable changes to the `algebraic-range` package.

## [0.9.0] — 2026-07-15

Complete rewrite from scratch, following the Wolfram Language
`AlgebraicRange` **2.0** source function by function
(`wolfram/AlgebraicRange-2-0-0.wl`). The Python test suite now transcribes
the WL 2.0 verification suite verbatim (91 tests), and the port reproduces
the WL output exactly on all functional tests.

### Changed — semantics (following WL 2.0)

- **Elementary ranges are anchored at the bounds, not at integers.**
  `algebraic_range(x, y)` now gives `Sqrt[Range[x², y²]]` exactly: the
  squared grid starts at `x²` with step 1. Previously the grid was rounded
  to integers, e.g. `algebraic_range(1/3, 3)` now starts `1/3, √10/3, …`
  instead of `1, √2, …`.
- **Approximate inputs are exactified** (the WL `realReplace` /
  `RootApproximant` step): decimal floats become exact rationals
  (`0.1 → 1/10`), and other floats are recognized as small-degree algebraic
  numbers when possible. E.g. `algebraic_range(0.1, 3.1)` now starts at
  `1/10`, where 0.8.x started at 1. This is the only behavioural
  disagreement with the retired 0.8 test suite.
- **The outer construction was redesigned** (WL 2.0 algorithm): output is
  equivalent to the documented `Outer` product but computed with a
  per-element window over the sorted factor grid — linear rather than
  quadratic complexity — and with a refined factor grid (`zero`-offset
  anchoring, exact endpoint handling).
- **Descending ranges** (negative step) are now first-class throughout,
  including mixed-sign bounds and the `"Root"` step method.
- **`farey_range=True` follows the improved `FareyRange`** resource
  function exactly: `FareyRange[a, b, n]` is the set of rationals with
  denominator ≤ n in `[a, b]`, generated by the Farey next-term recurrence;
  integer steps `s` are interpreted as order `s` (reciprocal steps `1/n` as
  order `n`), negative steps reverse, and other steps raise
  `FareyStepError`.
- **`formula_complexity`** now implements the WL 2.0 one-pass heuristic
  (radicals `b^(m/n)` contribute `|m|+|n|` copies of the base; rationals
  contribute numerator and denominator; built-in constants count as 1).
  Values differ from 0.8.x. *This heuristic is scheduled for a complete
  rewrite in a future version.*

### Added

- **`working_precision` option** (WL `WorkingPrecision`): sets the decimal
  precision of all internal numerical comparisons, enabling ranges with
  steps far below machine resolution (e.g. `10^-50` with
  `working_precision=60`).
- **New exceptions** mirroring the WL 2.0 failure objects:
  `FareyStepError` (bad step with the Farey option) and `LowerBoundError`
  (negative `d`); `StepBoundError` keeps signalling `d > |s|`.
- **Benchmark suite** (`benchmark/`): paired Python/Wolfram scripts over the
  five WL group-10 reference cases, with results in
  `benchmark/BENCHMARK.md`.
- Support for any numeric root order ≥ 1 in `root_order`.

### Deprecated

- `formula_complexity_threshold=` keyword: use `formula_complexity=`
  (matching the WL option name `"FormulaComplexity"`). The old keyword
  still works and emits a `DeprecationWarning`.

### Performance

- ~40×–100× faster than 0.8.2 where 0.8.2 completes at all; the stepped
  reference cases that 0.8.2 could not finish in 5 minutes now run in
  seconds — up to ~1.7× faster than Wolfram Language 2.0 itself.
  See `benchmark/BENCHMARK.md`.

### Known deviations from WL 2.0

- With machine precision and a step below double resolution (the regime the
  WL documentation warns about, prescribing `WorkingPrecision`), colliding
  neighbours are deduplicated against a single high-precision rounding of
  each value; WL's machine `N` may place a collision-class boundary a grid
  unit or two away, so the surviving representative can differ (WL suite
  case 9-a). With `working_precision` set, results match WL exactly
  (case 9-b).
- `RootApproximant` recovery of non-decimal floats is limited to algebraic
  numbers of degree ≤ 6 with moderate coefficients; beyond that the float
  is exactified to a rational.
